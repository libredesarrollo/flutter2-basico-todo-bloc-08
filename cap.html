El patrón BLoC en Flutter

En este capítulo construiremos una aplicación completa de gestión de tareas (Todo App) utilizando el patrón BLoC (Business Logic Component) en Flutter. A lo largo del proceso aprenderemos a separar la lógica de negocio de la interfaz de usuario, a persistir datos con la base de datos NoSQL Sembast y a manejar el estado de la aplicación con Streams. Al finalizar tendrás una base sólida para aplicar este patrón en proyectos reales de mayor complejidad.

Introducción al patrón BLoC

El patrón BLoC fue propuesto originalmente por el equipo de Google y se ha consolidado como uno de los enfoques de gestión de estado más utilizados en el ecosistema Flutter. Su nombre proviene de Business Logic Component y su idea central es simple pero poderosa: toda la lógica que no tiene que ver con la presentación visual debe vivir en una clase independiente, completamente desacoplada de los widgets.

La comunicación entre la capa de presentación y la capa de lógica de negocio se realiza a través de Streams y Sinks, primitivas reactivas incluidas en la librería estándar de Dart. Los widgets escuchan un Stream para reflejarse automáticamente cuando los datos cambian, y envían eventos al BLoC a través de un Sink cuando el usuario realiza alguna acción. Esta separación hace que el código sea más fácil de probar, de mantener y de escalar.

Creando el proyecto

Creamos un nuevo proyecto Flutter con el comando habitual. Al finalizar la creación limpiamos el código de ejemplo que genera Flutter por defecto, dejando únicamente el punto de entrada de la aplicación. La estructura de carpetas que adoptaremos separa cada responsabilidad en su propio directorio:

Este tipo de organización por capas facilita localizar cualquier pieza del sistema y es una convención ampliamente adoptada en proyectos Flutter profesionales.

Definiendo el modelo Todo

Antes de tocar la base de datos o la interfaz, definimos el modelo de datos. Una tarea (Todo) en nuestra aplicación tiene un nombre, una descripción, una fecha límite y un nivel de prioridad. El modelo también implementa métodos de serialización para comunicarse con Sembast, que almacena los registros como mapas de Dart. 

class Todo {
  int id;
  String name;
  String description;
  String completeBy;
  int priority;

  Todo(this.name, this.description, this.completeBy, this.priority): id=0;

  Todo.empty()
      : id = 0,
        name = "",
        description = "",
        completeBy = "",
        priority = 0;

  Map<String, dynamic> toMap() => {
        'name': name,
        'description': description,
        'completeBy': completeBy,
        'priority': priority,
      };

  static Todo fromMap(Map<String, dynamic> map) =>
      Todo(map['name'], map['description'], map['completeBy'], map['priority']);
}

Observa que el campo id no forma parte del mapa que se persiste. Sembast genera su propio identificador entero al insertar un registro, y lo recuperamos posteriormente asignándolo al campo id del objeto. El constructor nombrado Todo.empty() resulta muy útil cuando necesitamos un objeto vacío como punto de partida antes de que el usuario introduzca datos.

Instalando Sembast y path_provider

Sembast es una base de datos NoSQL embebida, orientada a documentos, diseñada específicamente para Flutter y Dart. No requiere un servidor externo ni configuración de esquemas: los datos se guardan en un archivo dentro del dispositivo. Para ubicar ese archivo en el directorio correcto de cada plataforma usamos path_provider, y para construir rutas de forma segura y multiplataforma utilizamos el paquete path.

Agregamos las dependencias en pubspec.yaml:

dependencies:
  flutter:
    sdk: flutter
  sembast: ^3.7.0
  path_provider: ^2.1.2
  path: ^1.9.0
  cupertino_icons: ^1.0.8

Después de guardar el archivo ejecutamos flutter pub get para que Flutter descargue e instale los paquetes.

La capa de acceso a datos: TodoDb

Con Sembast la unidad de almacenamiento equivalente a una tabla relacional se llama Store. Creamos un helper que encapsula todas las operaciones con la base de datos. La apertura de la base de datos se realiza de forma asíncrona; por eso en el método main() llamamos a initDB() antes de arrancar la aplicación. 

import 'package:path/path.dart';
import 'package:path_provider/path_provider.dart';
import 'package:sembast/sembast_io.dart';
import 'package:todoapp_08/models/todo_model.dart';

class TodoDb {
  static late Database _database;

  final todos = intMapStoreFactory.store('todos');

  initDB() async {
    final docsPath = await getApplicationDocumentsDirectory();
    _database = await databaseFactoryIo
        .openDatabase(join(docsPath.path, 'todoapp', 'sembast', 'todos.db'));
  }

  Future<Database> get database async {
    return _database;
  }

  Future insert(Todo todo) async {
    await todos.add(_database, todo.toMap());
  }

  Future<List<Todo>> getAll() async {
    final finder = Finder(sortOrders: [
      SortOrder('priority', false)
    ]);

    final todosSnapshot = await todos.find(_database, finder: finder);

    return todosSnapshot.map((snapshot) {
      final todo = Todo.fromMap(snapshot.value);
      todo.id = snapshot.key;
      return todo;
    }).toList();
  }

  Future delete(Todo todo) async {
    final finder = Finder(filter: Filter.byKey(todo.id));
    await todos.delete(_database, finder: finder);
  }

  Future update(Todo todo) async {
    final finder = Finder(filter: Filter.byKey(todo.id));
    await todos.update(_database, todo.toMap(), finder: finder);
  }

  Future<Todo?> getById(int id) async {
    final finder = Finder(filter: Filter.byKey(id));
    final snapshot = await todos.findFirst(_database, finder: finder);

    if (snapshot != null) {
      final todo = Todo.fromMap(snapshot.value);
      todo.id = snapshot.key;
      return todo;
    }

    return null;
  }
}

El campo estático _database se declara con late, lo que indica al compilador de Dart que la variable se inicializará antes de ser usada, aunque no lo sea en el instante de declaración. Esta es una adaptación necesaria al sistema de Null Safety introducido en Dart 2.12.

Operaciones CRUD

Las operaciones disponibles siguen el patrón clásico:

  • insert: agrega un nuevo registro usando todos.add().
  • getAll: obtiene todos los registros ordenados por prioridad de mayor a menor. El parámetro false en SortOrder indica orden descendente.
  • delete: elimina un registro filtrando por su clave primaria con Filter.byKey().
  • update: actualiza los campos de un registro existente. Se usa el mismo Finder con filtro por clave para apuntar al elemento correcto.
  • getById: busca un único registro. A diferencia de find() que devuelve una colección, usamos findFirst(). El resultado puede ser nulo si el identificador no existe, por lo que el tipo de retorno es Todo?.

Implementando el patrón BLoC

Con la capa de datos lista, llegamos al núcleo del capítulo: la implementación del BLoC. La clase TodosBloc es la responsable de orquestar todas las operaciones entre la base de datos y la interfaz de usuario.

Controladores y Streams

En Dart, un StreamController es el objeto que nos permite crear y gestionar un Stream. A través de su propiedad sink se introducen datos, y a través de su propiedad stream los widgets los escuchan. Necesitamos cuatro controladores: uno para emitir la lista completa de todos y otros tres para los eventos de inserción, actualización y eliminación. 

import 'dart:async';
import 'package:todoapp_08/helpers/todo_db.dart';
import 'package:todoapp_08/models/todo_model.dart';

class TodosBloc {
  late TodoDb db;
  List<Todo> todosList = [];

  // Stream para emitir la lista actualizada
  final _todosStreamController = StreamController<List<Todo>>.broadcast();

  // Controladores para cada operación de escritura
  final _todoInsertController = StreamController<Todo>();
  final _todoUpdateController = StreamController<Todo>();
  final _todoDeleteController = StreamController<Todo>();

  // Getters que exponen los Streams y Sinks al exterior
  Stream<List<Todo>> get todosStream => _todosStreamController.stream;
  StreamSink<List<Todo>> get todosAllSink => _todosStreamController.sink;

  StreamSink<Todo> get todoInsertSink => _todoInsertController.sink;
  StreamSink<Todo> get todoUpdateSink => _todoUpdateController.sink;
  StreamSink<Todo> get todoDeleteSink => _todoDeleteController.sink;
}

El controlador de la lista se crea con .broadcast() porque puede haber varios widgets escuchando al mismo tiempo el mismo Stream. Los controladores de escritura no necesitan broadcast porque solo hay un consumidor interno (el propio BLoC).

Los getters son la interfaz pública del BLoC. Los widgets solo pueden escuchar el Stream de la lista y escribir en los Sinks de operaciones; nunca tienen acceso directo a los StreamController. Esto es una práctica de encapsulamiento fundamental en este patrón.

El constructor y la lógica interna

En el constructor del BLoC inicializamos la base de datos, cargamos los datos iniciales y suscribimos cada controlador a su método correspondiente: 

TodosBloc._internal() {
  db = TodoDb();
  getAll();

  _todosStreamController.stream.listen(returnAll);
  _todoInsertController.stream.listen(insert);
  _todoUpdateController.stream.listen(update);
  _todoDeleteController.stream.listen(delete);
}

La llamada a getAll() en el constructor garantiza que en el momento en que cualquier widget empiece a escuchar el Stream ya tendrá datos disponibles. Cada llamada a listen() establece la conexión reactiva: cuando alguien añade un Todo al Sink de inserción, automáticamente se ejecuta el método insert(). 

Future getAll() async {
  List<Todo> todos = await db.getAll();
  todosList = todos;
  todosAllSink.add(todos);
}

List<Todo> returnAll(todos) {
  return todos;
}

void delete(Todo todo) {
  db.delete(todo).then((_) => getAll());
}

void update(Todo todo) {
  db.update(todo).then((_) => getAll());
}

void insert(Todo todo) {
  db.insert(todo).then((_) => getAll());
}

void dispose() {
  _todosStreamController.close();
  _todoInsertController.close();
  _todoUpdateController.close();
  _todoDeleteController.close();
}

El patrón que siguen insert, update y delete es idéntico: realizan la operación en la base de datos y cuando la promesa se completa (.then()) vuelven a llamar a getAll() para emitir una lista actualizada por el Stream. Así, la interfaz se sincroniza automáticamente sin que ningún widget tenga que solicitar la actualización de forma explícita.

El método dispose() cierra todos los controladores cuando la página que usa el BLoC es destruida. No hacerlo provocaría fugas de memoria.

El Singleton y el constructor Factory

Existe un problema sutil pero importante: si la página de listado y la página de guardado crean cada una su propia instancia de TodosBloc, estarán manejando Streams completamente diferentes. Los eventos que una página emite no llegarán a la otra, y la interfaz dejará de sincronizarse correctamente.

La solución es el patrón Singleton: garantizar que toda la aplicación comparta exactamente la misma instancia de TodosBloc. En Dart esto se implementa de forma elegante combinando una propiedad estática con un constructor factory. 

class TodosBloc {
  static final TodosBloc _singlenton = TodosBloc._internal();

  // ... propiedades y controladores ...

  factory TodosBloc() {
    return _singlenton;
  }

  TodosBloc._internal() {
    db = TodoDb();
    getAll();

    _todosStreamController.stream.listen(returnAll);
    _todoInsertController.stream.listen(insert);
    _todoUpdateController.stream.listen(update);
    _todoDeleteController.stream.listen(delete);
  }
}

El campo estático _singlenton se inicializa una única vez durante la vida de la aplicación mediante el constructor privado _internal(). El constructor factory es un tipo especial de constructor en Dart que permite retornar un valor en lugar de crear siempre un objeto nuevo. Cada vez que cualquier parte del código escriba TodosBloc(), el constructor factory simplemente devolverá la instancia ya existente.

Para verificar que el singleton funciona correctamente, podemos usar la función identical() de Dart, que retorna true si dos referencias apuntan exactamente a la misma posición de memoria: 

// En main(), antes de runApp():
print(identical(todosBloc, TodosBloc())); // imprime: true

Si el resultado es true, ambas variables son el mismo objeto. Si fuera false, estaríamos ante dos instancias distintas y el singleton no estaría funcionando.

La interfaz de usuario

Punto de entrada y rutas

La aplicación define dos rutas: la página de listado es la ruta inicial y la página de guardado se usa tanto para crear como para editar un Todo. 

void main() async {

  TodoDb todoDb = TodoDb();

class MyApp extends StatelessWidget {
  const MyApp({super.key});

  @override

      title: 'TodoApp',

        ListPage.route: (_) => const ListPage(),

La página de listado y el StreamBuilder

La página de listado es donde el patrón BLoC se hace visible de forma más evidente. El widget StreamBuilder escucha el Stream del BLoC y reconstruye automáticamente el ListView cada vez que llegan nuevos datos. 

class ListPage extends StatefulWidget {
  static const String route = "/";

  const ListPage({super.key});

  @override
  State<ListPage> createState() => _ListPageState();
}

class _ListPageState extends State<ListPage> {
  late List<Todo> todos;
  late TodosBloc todosBloc;

  @override
  void initState() {
    todosBloc = TodosBloc();
    super.initState();
  }

  @override
  void dispose() {
    todosBloc.dispose();
    super.dispose();
  }

  @override
  Widget build(BuildContext context) {
    todos = todosBloc.todosList;

    return Scaffold(
      appBar: AppBar(
        title: const Text("Listado"),
      ),
      floatingActionButton: FloatingActionButton(
        child: const Icon(Icons.add),
        onPressed: () {
          Navigator.pushNamed(context, SavePage.route);
        },
      ),
      body: StreamBuilder(
        stream: todosBloc.todosStream,
        initialData: todos,
        builder: (_, AsyncSnapshot snapshot) {
          return ListView.builder(
            itemCount: snapshot.hasData ? snapshot.data.length : 0,
            itemBuilder: (_, index) {
              return Dismissible(
                key: Key(snapshot.data[index].id.toString()),
                child: ListTile(
                  leading: CircleAvatar(
                    backgroundColor: snapshot.data[index].priority > 5
                        ? Colors.red
                        : Colors.purple,
                    child: Text(
                      "${snapshot.data[index].priority}",
                      style: const TextStyle(color: Colors.white),
                    ),
                  ),
                  title: Text("${snapshot.data[index].name}"),
                  subtitle: Text("${snapshot.data[index].description}"),
                  trailing: GestureDetector(
                    child: const Icon(Icons.edit),
                    onTap: () {
                      Navigator.pushNamed(context, SavePage.route,
                          arguments: snapshot.data[index]);
                    },
                  ),
                ),
                onDismissed: (_) {
                  todosBloc.todoDeleteSink.add(snapshot.data[index]);
                },
              );
            },
          );
        },
      ),
    );
  }
}

Hay varios aspectos importantes en este código:

  • StreamBuilder recibe el stream del BLoC y el parámetro initialData que se muestra mientras no hay ningún evento en el Stream. Esto evita que la lista aparezca vacía en el primer frame.
  • Dismissible es un widget de Flutter que permite deslizar un elemento para ejecutar una acción. La propiedad key debe ser única para cada elemento; usamos el id del Todo convertido en String. En el evento onDismissed añadimos el Todo al Sink de eliminación del BLoC.
  • El CircleAvatar cambia de color según la prioridad. Si es mayor que 5 aparece en rojo; si no, en púrpura. Esta es una forma sencilla de proporcionar retroalimentación visual al usuario sobre la urgencia de cada tarea.
  • Para navegar a la página de edición pasamos el objeto Todo como argumento de la ruta mediante Navigator.pushNamed(context, SavePage.route, arguments: snapshot.data[index]).

El widget reutilizable CustomTextField

En lugar de repetir la configuración de los campos de texto en cada formulario, extraemos esa lógica a un widget propio. Acepta como parámetros el texto de ayuda, el icono, el controlador, si el texto debe ocultarse y el tipo de teclado. 

import 'package:flutter/material.dart';

class CustomTextField extends StatelessWidget {
  final String placeholder;
  final IconData icon;
  final Color? primaryColor;
  final Color? accentColor;
  final bool obscureText;
  final TextEditingController controller;
  final TextInputType type;

  const CustomTextField({super.key,
    required this.placeholder,
    required this.icon,
    required this.controller,
    this.primaryColor,
    this.accentColor,
    this.obscureText = false,
    this.type = TextInputType.text,
  });

  @override
  Widget build(BuildContext context) {
    return TextField(
      obscureText: obscureText,
      controller: controller,
      keyboardType: type,
      decoration: InputDecoration(
        prefixIcon: Icon(icon),
        hintText: placeholder,
        contentPadding: const EdgeInsets.only(top: 14),
        enabledBorder: OutlineInputBorder(
          borderSide: BorderSide(
            color: primaryColor ?? Theme.of(context).primaryColor,
            width: 2.0,
          ),
        ),
        focusedBorder: OutlineInputBorder(
          borderSide: BorderSide(
            color: accentColor ?? Theme.of(context).colorScheme.secondary,
            width: 1.0,
          ),
        ),
      ),
    );
  }
}

Los parámetros de color son opcionales (Color?) y cuando no se proporcionan se toma el color primario y el color secundario del tema de la aplicación. Esto hace que el widget se adapte automáticamente a cualquier tema sin necesidad de configuración adicional.

La página de guardar: crear y actualizar en una sola pantalla

Una segunda página gestiona tanto la creación como la actualización de un Todo. La diferencia entre un modo y otro la determina el argumento que se pasa a la ruta: si es null la página muestra un formulario vacío para crear; si contiene un objeto Todo lo muestra precargado para editar. 

class SavePage extends StatelessWidget {
  static const String route = "/save";

  final TextEditingController nameEC = TextEditingController();
  final TextEditingController descriptionEC = TextEditingController();
  final TextEditingController completeEC = TextEditingController();
  final TextEditingController priorityEC = TextEditingController();

  final TodosBloc todosBloc;

  SavePage({super.key}) : todosBloc = TodosBloc();

  @override
  Widget build(BuildContext context) {
    final Todo todo = ModalRoute.of(context)!.settings.arguments == null
        ? Todo.empty()
        : ModalRoute.of(context)!.settings.arguments as Todo;

    if (todo.id != 0) {
      nameEC.text = todo.name;
      descriptionEC.text = todo.description;
      completeEC.text = todo.completeBy;
      priorityEC.text = todo.priority.toString();
    }

    return Scaffold(
      appBar: AppBar(
        title: Text(todo.id == 0 ? "Crear Todo" : todo.name),
      ),
      body: Card(
        child: Column(
          children: [
            CustomTextField(
              placeholder: "Nombre",
              icon: Icons.list,
              controller: nameEC,
            ),
            const SizedBox(height: 15),
            CustomTextField(
              placeholder: "Description",
              icon: Icons.description,
              controller: descriptionEC,
            ),
            const SizedBox(height: 15),
            CustomTextField(
              placeholder: "Completado",
              icon: Icons.check,
              controller: completeEC,
            ),
            const SizedBox(height: 15),
            CustomTextField(
              placeholder: "Prioridad",
              icon: Icons.star,
              type: TextInputType.number,
              controller: priorityEC,
            ),
            const SizedBox(height: 15),
            ElevatedButton(
              child: Text(todo.id == 0 ? "Insertar" : "Actualizar"),
              onPressed: () {
                todo.name = nameEC.text;
                todo.description = descriptionEC.text;
                todo.completeBy = completeEC.text;
                try {
                  todo.priority = int.tryParse(priorityEC.text) ?? 0;
                } catch (e) {
                  todo.priority = 1;
                }

                if (todo.id == 0) {
                  todosBloc.todoInsertSink.add(todo);
                } else {
                  todosBloc.todoUpdateSink.add(todo);
                }

                Navigator.pop(context);
              },
            ),
          ],
        ),
      ),
    );
  }
}

Varios detalles merecen atención:

  • SavePage es un StatelessWidget. El estado del formulario lo gestionan los TextEditingController, que se declaran como propiedades de la clase. En un StatelessWidget esto es posible siempre que no necesitemos llamar a setState().
  • La instancia de TodosBloc se asigna en el inicializador (: todosBloc = TodosBloc()). Gracias al Singleton, esta llamada devuelve exactamente la misma instancia que existe en la página de listado, por lo que los cambios se reflejan inmediatamente cuando el usuario regresa.
  • Al presionar el botón se determina si es una inserción o una actualización verificando si todo.id == 0. En el primer caso se añade al todoInsertSink; en el segundo, al todoUpdateSink.
  • Para la conversión de la prioridad usamos int.tryParse() combinado con el operador ??, que asigna 0 si el texto no es un entero válido. El bloque try/catch añade una capa adicional de seguridad.
  • Finalmente, Navigator.pop(context) regresa a la pantalla anterior y, como el BLoC ya emitió un nuevo estado a través del Stream, la lista se actualiza de manera reactiva en cuanto la página de listado vuelve a ser visible.

Adaptaciones al Null Safety

El código de este proyecto está escrito bajo las reglas de Null Safety de Dart, un sistema de tipos que permite al compilador detectar posibles errores de valor nulo en tiempo de compilación en lugar de en tiempo de ejecución.

Las adaptaciones más importantes realizadas en el proyecto son:

  • late: se usa para variables que se saben no nulas pero que no pueden inicializarse en el momento de la declaración. Ejemplo: static late Database _database en TodoDb y late TodosBloc todosBloc en el estado de la página de listado.
  • Tipos nullable con ?: el método getById declara su retorno como Future<Todo?> porque puede no encontrar ningún registro.
  • Operador !: al leer los argumentos de la ruta con ModalRoute.of(context)!.settings.arguments, el signo de exclamación le indica al compilador que asumimos que ModalRoute.of(context) no es nulo en ese punto. Si lo fuera, se lanzaría una excepción en tiempo de ejecución.
  • Color?: los parámetros de color en CustomTextField son opcionales, por lo que se declaran con tipo nullable y se resuelven con el operador ??.

Flujo completo de la aplicación

Vale la pena recorrer de un vistazo el ciclo de vida completo de una operación para interiorizar cómo todas las piezas encajan:

  1. El usuario pulsa el botón flotante en la lista y navega a SavePage.
  2. Completa el formulario y pulsa "Insertar".
  3. SavePage llama a todosBloc.todoInsertSink.add(todo).
  4. El BLoC recibe el evento a través del Stream de inserción y ejecuta insert(todo).
  5. insert() llama a db.insert(todo) y cuando finaliza llama a getAll().
  6. getAll() consulta la base de datos y emite la lista actualizada a través de todosAllSink.
  7. El StreamBuilder en ListPage recibe el nuevo evento y reconstruye el ListView automáticamente.
  8. Navigator.pop(context) regresa al listado donde el nuevo Todo ya es visible.

Este flujo unidireccional de datos es una de las grandes fortalezas del patrón BLoC: en todo momento sabemos de dónde vienen los datos y hacia dónde van. No hay mutaciones de estado dispersas por los widgets; toda la lógica fluye a través del BLoC.

Resumen

A lo largo de este capítulo hemos construido paso a paso una aplicación de tareas completa en Flutter, explorando cada capa del sistema:

  • Modelo de datos: la clase Todo con métodos de serialización para comunicarse con Sembast.
  • Capa de persistencia: la clase TodoDb que encapsula todas las operaciones CRUD con la base de datos NoSQL Sembast.
  • BLoC: la clase TodosBloc que orquesta la lógica de negocio a través de Streams y Sinks, implementada como Singleton mediante un constructor Factory para garantizar que todas las páginas comparten el mismo estado.
  • Interfaz reactiva: el uso de StreamBuilder para actualizar la lista automáticamente, Dismissible para eliminar tareas deslizando, y la navegación entre páginas pasando argumentos a través de las rutas.
  • Widgets reutilizables: el componente CustomTextField que centraliza el estilo de los campos de texto.

El patrón BLoC sin el paquete oficial flutter_bloc, tal como lo hemos implementado en este capítulo, es una excelente forma de comprender los fundamentos reactivos sobre los que ese paquete está construido. Una vez que entiendes cómo funcionan los Streams y Sinks de forma manual, usar cualquier librería de gestión de estado se vuelve mucho más intuitivo.
